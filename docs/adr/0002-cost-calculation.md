# ADR-0002: コスト計算のベース単価選定

## ステータス

Accepted

## コンテキスト

Infra-Miruのコスト最適化提案機能では、Kubernetesリソースの過剰プロビジョニングを金額換算する必要がある。
CPU（millicores）とメモリ（bytes）それぞれに月額単価を設定し、削減可能量と掛け合わせて月間節約額を算出する。

単価のベースとして以下の候補を検討した。

| 候補 | CPU単価 (¥/millicore/月) | メモリ単価 (¥/MB/月) | 特徴 |
| --- | --- | --- | --- |
| GCP e2-standard | 3.5 | 0.5 | 汎用インスタンス。日本リージョン実勢価格 |
| AWS m6i | 4.0 | 0.6 | 東京リージョン On-Demand |
| Azure Dv5 | 3.8 | 0.55 | Japan East On-Demand |
| カスタム入力 | ユーザー指定 | ユーザー指定 | 柔軟だが初期設定が面倒 |

## 決定

**GCP e2-standard シリーズの実勢価格をデフォルト単価**として採用する。

- CPU: ¥3.5 / millicore / 月
- Memory: ¥0.5 / MB / 月

APIレベルでカスタム単価の注入もサポートする（`NewCalculatorWithPricing`）。

## 理由

1. **中間価格帯**: 3大クラウド（AWS/GCP/Azure）の中でGCPは中間的な価格設定。過大でも過小でもない見積もりが可能
2. **日本円ベース**: 為替変動を受けずに直感的な金額を提示できる。ユーザーの大半が日本国内運用を想定
3. **汎用インスタンス基準**: e2-standardは最も一般的なワークロードで使われる汎用シリーズ。特殊用途（GPU、ハイメモリ等）は対象外とすることで計算をシンプルに保つ
4. **カスタマイズ可能**: `NewCalculatorWithPricing` により任意の単価を注入可能。AWS/Azure環境やRI（Reserved Instances）適用後の単価にも対応できる
5. **MVP適合**: 初期リリースでは精緻な見積もりよりも「どの程度のムダがあるか」の気づきを提供することが重要。デフォルト値で十分

### 不採用理由

- **AWS m6i**: 最高価格帯のため、節約額が過大に見えるリスク
- **カスタム入力のみ**: 初回利用時のハードルが高い。デフォルト値がないとユーザーが離脱する可能性

## 影響

- `cost.Calculator` はデフォルトでGCPベース単価を使用する
- 将来的にクラウドプロバイダーごとのプリセット追加が可能（`NewCalculatorForAWS` 等）
- 金額は概算であり、実際の請求額とは差異がある旨をUIに表示すべき
